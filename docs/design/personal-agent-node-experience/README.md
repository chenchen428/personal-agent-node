# Personal Agent Node 与 Cloud 四端体验设计

状态：待产品负责人视觉与交互验收
设计范围：PA 桌面端、移动端网页、Cloud 客户门户、Cloud 管理后台；不包含现有产品实现修改
可交互设计稿：[prototype.html](./prototype.html)

## 1. 设计结论

Personal Agent 不再用一套响应式页面承担所有目标，而是按运行位置、权限和用户意图拆分为四个明确产品面：

| 产品面 | 用户目标 | 核心内容 | 明确排除 |
| --- | --- | --- | --- |
| PA 桌面端 | 看清这台电脑上的 PA 正在做什么、拥有什么、如何连接 | 移动端全部内容，以及对话记录、Worker 详情、只读邮件、Online Pages、本机数据查看、渠道、技能目录、初始化和运行设置 | 手工创建 Worker、邮件处理动作、数据库写入/快照、技能添加/配置、外部网页登录 |
| 移动端网页 | 随时知道 PA 在做什么，并只读结果 | 最近动态、对话记录、Worker 详情、邮件、Online Pages、技能摘要 | 写入和审批、本机路径、CLI、数据库、底层运行日志、仅本机可执行的管理操作 |
| Cloud 客户门户 | 看懂自己的 Cloud 身份、域名、Node 连接与 Edge 配额 | Owner 控制台、托管/自定义域名、Node 授权与安装、公共服务状态 | 本机对话、邮件正文、附件、数据库、Agent 调度、计费购买 |
| Cloud 管理后台 | 运营独立的 Cloud 控制面 | Owner/站点、域名、Node 凭据状态、资源策略、运行开关、脱敏审计 | 客户本机内容、Owner 会话复用、Agent 工作区、模型网关与计费 |

四端共享 `PA` 品牌语言、颜色与状态定义，但不共享权限或页面实现。`Node Client` 不再作为用户名称，统一称为 **PA 桌面端**。移动端是只读活动流；Cloud 客户门户只解释连接资源；Cloud 管理后台使用独立 Host、独立管理员凭据与 8 小时会话。移动端出现的 Node 内容在桌面端都能查看，反过来不成立。

## 2. 不可破坏的产品边界

### 2.1 桌面套壳边界

1. 套壳 WebView 只展示 Personal Agent 本机客户端页面。
2. 外部网页、登录、OAuth、设备授权和外部帮助文档统一交给系统默认浏览器。本机对话、Worker、邮件、数据和 Online Page 详情属于客户端页面，直接在套壳内打开。
3. 客户端发起外部操作前显示清晰的“将在默认浏览器继续”交接；回到客户端后自动刷新授权结果。
4. 不依赖套壳 WebView 的 Cookie、浏览历史、密码管理器或已有登录态。
5. 默认生命周期为：打开客户端启动本机服务；关闭客户端停止 Agent、客户端页面和穿透服务。存在运行中任务时，关闭前必须说明影响并请求确认。

### 2.2 移动端穿透边界

移动端不是 local 页面。没有可用穿透服务时，就没有移动端产品体验。

允许的移动入口示例：

```text
https://<user-tunnel-domain>/app/mobile
https://<user-tunnel-domain>/app/mobile/conversations/<session-id>
https://<user-tunnel-domain>/app/mobile/workers/<worker-id>
https://<user-tunnel-domain>/pages/<publication-slug>
```

禁止向用户表达的地址：

```text
http://localhost:*
http://127.0.0.1:*
http://[::1]:*
http(s)://*.local/*
file://*
客户机绝对路径或工作区内部路径
```

Agent、worker、skill 和 Harness 产生的用户可见链接必须先经过统一的 `mobile-safe URL` 转换。无法转换的本机产物不能显示为手机可打开链接，应显示“仅可在电脑上使用”，并提供在桌面客户端中定位的动作。

## 3. 桌面客户端信息架构

桌面导航分为 Workspace 和 System 两组：

| 入口 | 回答的问题 | 主要组件 |
| --- | --- | --- |
| 总览 | 现在能不能用，还需要我做什么？ | 就绪度、异常优先状态、必要操作、近期系统事件 |
| 对话记录 | 我通过各渠道和 PA 沟通过什么？ | 渠道身份、对话主题、消息、真实计划、关联 Worker |
| Workers | PA 自动启动了哪些任务，现在做到哪里？ | 来源对话、运行状态、计划步骤、最近事件、最终输出 |
| 邮件 | PA 的邮箱地址是什么，收到了什么？ | 邮箱说明、本机收件箱、正文、附件 |
| Online Pages | PA 发布了什么？ | 公开/私有筛选、本机详情、可见性、来源对话、可选配图 |
| 数据 | PA 为什么保存这些结构化数据？ | 数据库树、列结构、只读 SQL、表格结果 |
| 渠道 | 微信、邮箱、平台和手机网页分别怎样连接？ | 连接方式、数据边界、托管/自定义域名流程 |
| 初始化设置 | 我怎样完成第一次配置？ | 结果导向的步骤清单、默认浏览器授权交接 |
| 技能 | PA 可以做什么，权限是什么？ | 只读能力目录、用途、可用状态和权限摘要 |
| 运行设置 | 何时启动/停止，失败后怎么办？ | 生命周期、自动恢复、更新与回退预期 |

### 3.1 总览模板

总览只使用三层信息：

1. **当前结论**：例如“你的 Personal Agent 已经可以工作”。
2. **需要行动的差距**：例如“只剩微信连接，约 2 分钟”。
3. **可解释状态**：Agent、穿透、渠道三个状态足以覆盖日常判断。

不在总览展示访问量、Online Pages 数量、token 用量等无行动价值的数据。健康状态不堆叠成大量 KPI 卡；异常发生时，对应状态卡提升优先级并出现单一修复动作。

### 3.2 初始化步骤模板

```text
[步骤结果]
一句话说明为什么需要
当前状态：未开始 / 进行中 / 已完成 / 需要处理
唯一主动作：开始 / 继续 / 修复 / 查看
```

核心顺序：

1. 建立本机管理身份。
2. 选择 Agent 运行方式和工作区。
3. 通过客户端二维码连接微信。

邮箱与手机网页放在“可选能力”层：邮箱解释公网身份、用户自管转发/MTA 与本机保存边界；手机网页解释托管域名、自定义域名和穿透。两者不参与核心 Agent 是否可工作的同级进度。

### 3.3 默认浏览器交接模板

交接弹层只回答三件事：

- **去哪儿**：将在默认浏览器打开的站点域名。
- **为什么**：客户端不承载外部登录，也不共享浏览器 Cookie。
- **如何回来**：完成后返回客户端，状态会自动刷新。

禁止在套壳中嵌入授权页，也禁止用通用 WebView 打开任意 URL。

### 3.4 对话记录与 Worker

用户不需要理解“主 Agent 会话”“主体会话”或内部 session。一级菜单改为“对话记录”，只展示用户通过微信或桌面端直接与 PA 沟通的内容，并按渠道身份、主题和时间组织。内部可复用 session 仍是实现事实，但不作为用户概念。

界面区分规则：

- 直接沟通使用“微信对话 · 用户”或“桌面端对话”标签，不使用“主 Agent”。
- Worker 由 PA 自动调度，不提供“新建 Worker”。
- Worker 始终显示来源对话、启动原因、运行状态、当前计划步骤、最近可解释事件和最终输出。
- Worker 同时显示过程中已经向用户表达的阶段性回复，例如“保留 7 个候选，正在核对两个预约入口”。这些是可读进展，不是内部 reasoning、prompt、工具参数或原始日志。
- 对话中的 Worker 是可点击的关联任务卡；点击进入桌面 Worker 详情。
- Worker 完成时，结果回到来源对话；不伪装成普通 PA 消息，也不展示内部 prompt。
- 不根据工具调用数或时间估算完成百分比。

### 3.5 邮件

邮件是只读查看功能。页面首先展示 PA 邮箱地址，并明确告诉用户：可以把资料、通知或任务邮件发到这个地址，PA 会在本机收到。主体是可点击的邮件列表、正文和附件；不提供“交给主 Agent”“标记已读”“回复”或其他处理按钮。

公网邮箱身份可以随 Cloud 资源授权获得，但 Personal Agent 不托管 SMTP/IMAP。真实邮件由用户配置的转发或自管 MTA 投递到 Node，正文和附件只保存在本机 Workspace。

### 3.6 Online Pages

桌面端是 Online Pages 的页面库与本机阅读面：

- 按全部、私有、公开筛选。
- 显示标题、摘要、更新时间、来源对话或 Worker。
- 私有页面走已认证 `/publications/*` 路径；公开页面走 `/pages/*`、`/blog/*` 等公开路径。
- 客户端点击“查看”进入套壳内的本机详情路由；不会触发默认浏览器交接。
- 移动端点击后进入同一穿透域名或已审核公开域名下的详情页。
- 列表首图优先使用 PA 生成的主题 SVG，并提供替代文本；不是只写几个标题字。没有合适首图时使用稳定的 PA 视觉模板，不产生破图或空白大块。
- Online Page 正文是 Agent 交付的任意 HTML，不是由宿主按文章 schema 重新渲染。详情页只保留返回、可见性和路径等最小宿主外壳，其余区域完整交给页面 HTML。

现有 Node 已分别具备公开页面上传和私有 publication store，但当前 UI 仍需把两套索引合并为统一页面库。设计批准后，Online Pages 发布技能需要增加配图合同：优先生成轻量 SVG，也允许用户资产；外部图片必须保留来源与使用许可，所有图片必须生成替代文本。

### 3.7 PA Data Sheets

数据只进入桌面客户端，不进入移动端。它不是数据库运维中心，而是一个高密度、只读的 Sheet 查看器，让用户直接看到“PA 使用了哪些结构化数据”。模块价值融入页面导语，不再单独占用说明卡：

- 标题右侧用一个紧凑统计区展示托管方、工作表数量、数据库体量与只读权限，不占用表格垂直空间。
- 页面主体是一整块 Sheet 工作台：工具栏、公式/字段说明栏、筛选条件、列字母、行号、字段表头、密集数据网格、工作表标签和分页。
- 每个数据库表或视图映射为一个工作表标签；底部直接切换 `activities`、`activity_audit`、`mail_messages`、`page_publications` 等数据集，不再使用左侧数据库树。动态由主 Agent 组织，审计表只保存脱敏操作证据。
- 字体使用 Arial / Microsoft YaHei 等适合密集表格的无衬线字体；29px 行高、固定列宽、粘性列头和水平滚动保证信息密度。
- 字段名、类型和用户可读说明显示在列头；选中单元格后，公式栏显示单元格坐标和完整字段说明。Agent 建表或变更字段时必须维护 `字段名 + 类型 + 描述`，客户端不猜测缺失含义。
- 筛选沿用 Excel 心智模型：从字段表头或“添加筛选”选择字段、运算符和值；多列筛选叠加，条件以可移除 chip 明示。后台生成查询，客户端不出现 SQL 编辑器。
- 默认每页 50 行，可切换 100 / 200 行。服务端只返回当前页和计数；网格只渲染可见行，避免大表造成 DOM 和内存压力。
- 数据库写入、迁移、备份、恢复、建表和字段修改继续由 PA、数据服务和现有权限合同托管。

UI 不显示数据库绝对路径，只显示可解释的工作表名称、大小、工作表数量和只读数据。

### 3.8 渠道连接

- **微信**：客户端生成一次性二维码，用户在微信中扫码并等待确认。
- **PA 邮箱**：Cloud 授权提供公网身份；邮件通过用户转发或自管 MTA 进入 Node，不承诺托管 SMTP/IMAP。
- **手机网页 / Online Pages**：依赖穿透服务。托管域名通过默认浏览器授权后自动分配；自定义域名依据 Cloud 代码采用“输入域名 → 配置 `_personal-agent` TXT 验证记录 → Cloud 查询验证 → 启用”的流程。
- **可选平台**：按扫码、验证码、过期和能力范围解释，不与 PA 核心就绪状态混为一谈。

独立“远程访问”一级菜单取消。手机访问、穿透和域名配置统一属于渠道；总览只显示状态，初始化只把它作为可选扩展。

### 3.9 技能

技能页面是只读能力目录。桌面端和移动端都不提供添加、配置、启停或编辑入口，只展示技能名称、用途、可用状态和用户能理解的权限摘要。

## 4. 移动端信息架构

移动端采用顶部标题 + 左侧抽屉菜单，不使用底部菜单。原因不是单纯视觉偏好，而是网页会与 Safari/Chrome 底部工具栏、安全区和滚动手势竞争。

侧边菜单建议顺序：

1. 最近动态
2. 对话记录
3. 运行中的 worker
4. 邮件
5. Online Pages
6. 技能

抽屉底部不重复“安全隧道在线”。页面可访问本身已经说明链路可用，底部改为“今天新增动态、运行中的 Worker、未读邮件”摘要，并提醒任务沟通继续在微信完成。

### 4.1 最近动态

最近动态是移动端默认首页。它合并不同来源，但使用统一项目结构：

```text
[活动类型] [相对时间]
可扫描标题
最多两行摘要
未读 / 进行中 / 已完成状态
```

支持的活动类型：

- 收到邮件和从邮件提取出的待办。
- 用户通过渠道发出的新指令。
- Agent 开始、暂停、完成或失败的任务。
- Agent 创建 worker、worker 阶段变化和完成。
- Online Page、报告或其他用户产物生成/更新。
- 外部写入或高风险操作的等待状态，但不在移动页面提供审批按钮。

列表按更新时间倒序。筛选只保留“全部 / 进行中 / 已完成”，每项带近似数量。邮件的“全部 / 未读 / 有附件”和其他有价值的筛选也统一显示计数。如果 PA 等待用户决定，动态仅说明“请回到微信对话继续”，不把移动端变成第二个审批中心。

动态、对话、Worker、邮件和 Online Pages 列表采用分批加载：首批 20 条，接近列表末尾时自动请求下一批，原型用“继续向下加载 / 已显示 X / Y”明确表现该行为。超过 100 个已加载项目后启用可见项渲染，保留滚动位置但不让所有历史节点长期留在 DOM 中。

### 4.2 对话记录与 Worker 进度

移动端对话不是内部事件日志。默认只显示用户消息、Agent 对用户的表达、关键 worker 状态与最终产物。

worker 详情采用四层：

1. 任务标题与父对话。
2. 当前阶段，例如“正在交叉核验”。
3. 阶段时间线：已完成、当前、等待。
4. 结果或下一次更新时间。

工具调用、内部 prompt、token、原始 stdout/stderr 和客户机路径不得直接暴露。失败时显示用户能理解的影响、是否自动重试和可执行动作。

移动端不出现“主 Agent 会话”概念。直接沟通明确标为“微信对话”或“桌面端对话”；Worker 标为“Worker”，同时显示来源对话。对话中的 Worker 以关联任务卡出现。

### 4.3 Online Pages

Online Pages 在桌面端是页面库和本机阅读面，在移动端是重要的只读结果阅读面，本质上也是 PA 的页面与博客。发布由 PA 在任务中完成，不在当前客户端设计手工发布表单。

- 列表以标题、摘要、更新时间和来源对话为主。
- 点击后始终在同一穿透域名或已审核的公开域名打开。
- 私有 Page 继承移动端登录和访问控制；公开 Page 必须明确标记公开范围。
- 从 Agent 对话进入 Page 时，不得生成 localhost 回退链接。

### 4.4 技能

移动端和桌面端都只读技能名称、用途、可用状态和用户可理解的权限摘要。配置、启停和编辑不进入当前客户端体验，包括：

- 工作区绝对路径和文件系统权限。
- Harness 注册表、内部路由、端口和 provider 密钥。
- 原始配置文件编辑。
- 会导致服务重启、数据迁移或不可逆外部写入的操作。

## 5. Cloud 信息架构与代码边界

### 5.1 Cloud 客户门户

客户门户基于 Cloud 现有 `/console`、`/connect`、`/download`、`/docs` 与 `/status` 能力整理为四个页面：

1. **我的 PA**：Owner 身份、托管域名、Node 心跳与设备摘要、今日 Edge 流量和有效策略来源。
2. **域名与访问**：托管域名结果、自定义域名 TXT 验证三步、穿透与可见性边界。
3. **连接与安装**：下载桌面端、默认浏览器中的一次性 Owner 授权、回到桌面端读取结果、已连接设备。
4. **服务状态**：只展示客户能理解的登录、Edge、域名/DNS 状态，不泄漏基础设施细节。

Cloud 客户门户不读取 Node 本机内容。Agent 邮箱在 Cloud 中只是地址身份；`mailOperational` 只有 Node 邮件能力实际可用时才可显示为可收件。

### 5.2 Cloud 管理后台

管理后台基于现有 `/admin`、`/admin/accounts`、`/admin/policies`、`/admin/settings` 与 `/admin/audit` 设计：

- 运行总览：Owner、在线 Node、域名异常、Edge 请求与出站流量。
- Owner 与站点：托管域名、Agent 邮箱身份、心跳、有效策略；重命名域名、站点级配额覆盖、撤销/恢复 Node 都要求原因和确认。
- 资源策略：`edge_daily_bytes`、`concurrent_connections`、`requests_per_second`、`max_body_bytes`、`max_stream_seconds`，明确 Default / Pilot / Site override 来源。
- 运行开关：暂停新 Free Owner、Managed Edge kill switch、账户上限、每日新 Owner 上限与 Edge 总预算。
- 脱敏审计：actor、action、target 与脱敏 metadata；不记录凭据、Cookie 或客户内容。

管理员后台使用独立 Host、独立 Cookie 与独立环境凭据，8 小时会话，不复用 Owner 登录。Cloud 当前不设计公开注册、付费托管、Billing、模型 token 试用或模型网关，这些均未出现在原型中。

## 6. 集合加载与分页合同

任何可能随使用时间持续增长的集合都不得一次性完整加载。原型和后续实现统一采用以下合同：

| 场景 | 交互 | 默认批量 | 实现要求 |
| --- | --- | ---: | --- |
| 桌面对话、Worker | 数字分页 | 20 | 服务端分页；搜索或筛选变化时回到第 1 页 |
| 桌面邮件 | 数字分页 | 50 | 服务端分页；显示范围、总数和当前页 |
| 桌面 Online Pages | 数字分页 | 12 | 可切换 12 / 24 / 48；公开/私有筛选重新计数 |
| PA Data Sheets | 数字分页 | 50 | 可切换 50 / 100 / 200；服务端查询与可见行渲染 |
| Cloud Owner / 管理后台增长集合 | 数字分页 | 25 | Owner、设备、策略、审计分别分页；操作始终独立成列 |
| 移动端动态、对话、Worker、邮件、Pages | 游标分批加载 | 20 | 接近末尾预取；显示 `已显示 X / Y`；失败时原位重试 |

分页请求必须携带稳定排序键和游标或 `page/pageSize`，响应至少包含 `items`、`total`、当前范围和是否还有下一批。筛选、搜索、页大小变化会取消旧请求、重置到首批并避免重复项；刷新保持当前筛选，但回到第一页。加载态使用与行高一致的骨架行，错误只替换分页区域，不清空已加载内容。

固定、规模有明确上限的配置集合不强行分页，例如客户门户的“资源与入口”、服务状态和运行开关。它们仍必须使用明确列网格；有操作的表格必须提供独立“操作”列，无操作行显示 `—`，避免状态和值因按钮有无而错位。

## 7. 交互参考与取舍

- Excel 官方把筛选放在列标题，支持文本/数字运算符、多列叠加、清除与重新应用；本设计用字段筛选和可移除条件 chip 复用这一心智模型，后台再生成查询。[Microsoft Support: Filter data in a range or table](https://support.microsoft.com/en-US/Excel/get-started/filter-data-in-a-range-or-table-in-excel)
- AG Grid 的服务端模型以 `startRow` / `endRow` 只请求当前页，并由服务端返回总行数；桌面和 Cloud 的增长集合沿用此分页边界。[AG Grid: Server-Side Pagination](https://www.ag-grid.com/javascript-data-grid/server-side-model-pagination/)
- Handsontable 的行虚拟化只渲染可见行，其分页插件提供页大小、页码和导航状态；PA Data Sheets 将这两项作为大表性能合同，而不是要求实现必须采用某个具体组件。[Handsontable: Row virtualization](https://handsontable.com/docs/9.0/row-virtualization/)；[Handsontable: Rows pagination](https://handsontable.com/docs/react-data-grid/rows-pagination/)

## 8. 设计稿交付质量门禁

每轮设计稿交付前必须逐页走查全部改动页面和重要状态：

- 桌面/Cloud 代表宽度：至少 1440 px；移动端：至少 390 × 844。
- 对齐与网格、文字对比与字号、截断与溢出、按钮间距、唯一激活状态。
- 列表/详情、筛选、菜单、返回、弹层和所有宣称可点击的入口必须有响应。
- 覆盖正常、空、加载、失败或受限状态；原型未实现的状态必须在文档中说明。
- 浏览器走查属于交付前自检，不等同于用户视觉验收，也不能授权产品实现。

本轮 V5 走查记录：

- 静态交互合同已检查：4 个顶层验收面；桌面 11、移动 10、Cloud 客户 4、Cloud 管理 5 个页面目标均有对应面板，没有重复 `id`，且所有导航与详情入口都有对应面板。
- 内联脚本已通过 JavaScript 语法检查；原型中没有 CNAME 旧文案、SQL 操作控件、Cloud Billing 假能力或 localhost/file 用户链接。
- 响应式规则已覆盖 900 px 和 600 px 断点；桌面/Cloud 导航折叠、四端顶栏滚动、Sheet 横向滚动、数据统计折行和 Cloud 表格降级均有明确规则。
- 尝试接管当前 `file://` 原型进行浏览器点击与截图时，被浏览器 URL 安全策略拒绝。未绕过限制，因此真实像素级视觉走查和截图仍标记为待用户本地验收，不能由上述静态检查替代。

## 9. Codex CLI 能力依据

本轮设计以本机 `codex-cli 0.139.0`、其生成的 App Server protocol schema 和 Node 的 `app-server-mapper.ts` 为依据。App Server 命令本身仍标记为 experimental，因此 UI 只使用 Node 已经适配并测试的事件。

| UI 表达 | 当前依据 | 设计结论 |
| --- | --- | --- |
| To Do / 计划步骤 | `turn/plan/updated` 带完整 step/status；plan item 有最终文本 | 可以显示真实 checklist 和当前步骤 |
| 对话 / Worker | Node `role`、`parentSessionId`；Codex `parentThreadId`、sub-agent/collab 状态 | 内部仍可关联父子 session；用户界面表达为来源对话和 Worker 任务 |
| 最近进展 | command execution、file change、MCP tool call、Web Search、turn 状态 | 只显示经过翻译的最近事件 |
| 文件变更 | 文件列表、diff 和增删行统计 | 桌面端可查看；移动端只显示结果摘要 |
| 引用 | Agent 回复中的 Markdown 链接与受管产物 URL 可读取；通用结构化 citation 尚未由当前 Node mapper 稳定暴露 | 不设计万能“引用面板”，只展示实际回复链接和已登记产物 |
| 完成百分比 | 协议没有可靠百分比 | 禁止显示 68% 等估算，改用计划步骤和运行状态 |
| 请求用户输入/授权 | App Server 能产生结构化问题和 approval 请求 | 桌面端可承载；移动端只读并引导回微信对话 |
| Goal | 当前 CLI feature 为 stable，Node runner 支持 goal get/set/clear | 可作为高级会话状态，暂不抢占普通用户界面 |

用户界面不直接显示 reasoning、原始命令输出或内部工具参数。所谓“进度”是 `计划步骤 + 当前运行状态 + 最近可解释事件`，不是模型思考直播。

## 10. 视觉系统

设计继续遵循现有 `DESIGN.md` 的暖色编辑风，但桌面端从营销式大版面调整为工作型 SaaS 密度。

| Token | 值 | 使用 |
| --- | --- | --- |
| Canvas | `#faf9f5` | 桌面主背景、浅色页面 |
| Ink | `#141413` | 主文字、深色导航 |
| Coral | `#cc785c` | 唯一主操作、当前进度、需要关注 |
| Surface | `#f5f0e8` | 次级区块、移动图标底色 |
| Card | `#efe9de` | 引导说明、抽屉选中态的暖色对应面 |
| Hairline | `#e6dfd8` | 表格、列表和输入分隔 |
| Success | `#4f9d6a` | 在线、完成 |
| Warning | `#c28b26` | 部分可用、等待配置 |
| Error | `#c64545` | 失败、阻塞、必须处理 |

字体使用 Cormorant Garamond/宋体类显示字体承载少量页面标题，Inter/系统无衬线字体承载所有工作 UI。桌面大标题控制在 44px 以内，移动标题控制在 24px 左右；列表和状态页不使用 hero 字号。

组件原则：

- 主操作每个区域最多一个珊瑚色按钮。
- 卡片圆角不超过 12px，操作控件不超过 8px。
- 页面区块默认无悬浮卡片，只有状态实体、弹层和有边界的工具使用卡片。
- 状态必须同时使用文字与颜色，不依赖颜色单独表达。
- 移动活动流使用分隔线，不把每一项做成独立卡片。
- 图标使用统一线性图标集；正式实现优先使用现有 Lucide。

## 11. 核心状态

### 10.1 桌面生命周期

```text
启动客户端
  → 启动/连接本机服务
  → 验证当前不可变版本
  → 启动穿透（若已配置）
  → 进入总览

关闭客户端
  → 检查运行中任务
  → 必要时请求确认
  → 停止穿透
  → 停止本机服务
  → 关闭窗口
```

### 10.2 移动入口

| 状态 | 移动端体验 |
| --- | --- |
| 穿透在线、已登录 | 正常进入最近动态 |
| 穿透在线、未登录 | 在穿透域名完成认证后返回原目标页 |
| 客户端正在启动 | 显示轻量等待页，并有限次数自动刷新 |
| 穿透中断 | 显示“这台 Node 暂时离线”，不回退到 local 地址 |
| 客户端已关闭 | 显示稳定的离线页；说明需要在电脑打开 Personal Agent |
| 会话过期 | 保留原目标路径，重新登录后继续 |

## 12. Harness 与技能输出合同

所有 Node Harness 和 skill 的用户表达应采用以下产物模型：

```json
{
  "title": "北海道 7 日家庭行程",
  "kind": "online-page",
  "status": "ready",
  "localResource": "/workspace/pages/hokkaido-family.html",
  "mobileUrl": "https://<tunnel-domain>/publications/hokkaido-family",
  "visibility": "private",
  "cover": {
    "kind": "generated-svg",
    "resource": "/workspace/pages/assets/hokkaido-family-cover.svg",
    "alt": "北海道家庭行程标题封面"
  },
  "sourceSessionId": "session-id"
}
```

表达规则：

1. 给用户的回复优先使用 `mobileUrl`。
2. `localResource` 只用于 Node 内部解析，不直接进入对话。
3. 未配置穿透时，Agent 不伪造 URL，改为说明“移动访问尚未开启”，并给出桌面客户端的设置入口名称。
4. worker 产生中间产物时，只有已注册且通过访问控制的资源可生成移动链接。
5. 验收测试应扫描用户可见消息、活动、邮件摘要和 Page 元数据，拒绝 localhost、`.local`、`file://` 与绝对路径泄漏。
6. `cover` 可缺省。存在时优先使用生成 SVG 或用户资产；外部图片需记录来源与许可，并始终提供 `alt`。缺省时由页面模板生成 PA 标题封面。
7. Online Page 的 `localResource` 指向完整 HTML；客户端详情和移动端详情以受控容器加载它，不把正文降级为结构化文章 schema。
8. 数据技能维护的每个可展示字段必须包含稳定字段名、数据库类型和用户可读说明；缺失说明时标记“尚未描述”，不得由客户端编造含义。
9. 所有增长集合必须通过稳定排序键与服务端分页或游标分批读取；不得把完整历史记录一次性注入页面。筛选、搜索和页大小变化必须重置分页状态并取消过期请求。

## 13. 验收清单

### 桌面客户端

- [ ] 首屏能在 5 秒内回答“是否可用”和“我还要做什么”。
- [ ] 初始化设置的每一步只有一个明确主动作。
- [ ] 外部登录和普通网页都由默认浏览器打开。
- [ ] 返回客户端后能更新授权结果，不要求用户重复输入。
- [ ] 关闭客户端时，用户能预期本机服务和移动入口会停止。
- [ ] 移动端能查看的对话、Worker、邮件、Pages 和技能在桌面端也都能查看。
- [ ] “对话记录”只展示用户与 PA 的直接沟通；Worker 作为可点击关联任务，不提供手工新建。
- [ ] Worker 桌面详情包含来源对话、启动原因、真实计划、阶段性用户可见回复和最终输出。
- [ ] 邮件页面先解释邮箱地址和用途，并且只提供列表、正文与附件查看。
- [ ] Online Pages 支持公开/私有筛选，列表首图为有主题的 SVG；客户端“查看”进入由通用 HTML 占据主体的本机详情。
- [ ] 数据页以高密度 Sheet 为主体，统计位于标题右侧；提供工作表标签、字段说明、通用筛选、排序、服务端分页和可见行渲染，不出现 SQL、写入或快照按钮。
- [ ] 对话、Worker、邮件和 Online Pages 都显示页码、总量、当前范围与每页条数，不一次性加载全部历史数据。
- [ ] 渠道页分别说明微信二维码、PA 邮箱投递、托管域名和自定义域名流程。
- [ ] 技能页只读展示，不出现添加、配置或启停操作。
- [ ] 800px 以上窗口宽度没有文字或控件重叠。

### 移动端

- [ ] 首屏是按时间排序的 Agent 活动，不是桌面仪表盘缩放版。
- [ ] 点击活动可进入邮件、对话、worker 或 Page 详情。
- [ ] 动态、对话、Worker、邮件和 Pages 显示“已显示 X / Y”，接近末尾加载下一批，加载失败不清空已有内容。
- [ ] worker 进度能区分已完成、当前和等待阶段，并显示 Worker 的阶段性用户可见回复。
- [ ] 不出现协议并未提供的虚构完成百分比。
- [ ] 使用侧边菜单，无底部菜单。
- [ ] 侧边菜单任意时刻最多一个入口为激活状态。
- [ ] 最近动态只提供带计数的全部、进行中和已完成筛选，不提供处理/审批入口。
- [ ] Online Pages 可区分公开和私有，并进入正文阅读页。
- [ ] 浏览器底部工具栏和安全区不会遮挡主操作。
- [ ] 所有可打开链接来自穿透域名或已审核公开域名。
- [ ] 穿透不可用时不出现 localhost 或 `.local` 回退。
- [ ] 320px 宽度下最长标题、域名和按钮文字不溢出。

### Cloud 客户门户

- [ ] 只展示 Owner、域名、Node 连接、Edge 配额、安装授权与公共状态，不出现 Node 本机内容。
- [ ] 托管域名和自定义域名流程分开，自定义域名使用 TXT 验证而不是 CNAME 文案。
- [ ] Agent 邮箱地址身份与实际可收件状态明确区分。
- [ ] 浏览器一次性授权、回到桌面端读取结果和设备状态能形成闭环。

### Cloud 管理后台

- [ ] 独立 Host、Cookie、管理员凭据与 8 小时会话边界清晰，不复用 Owner 身份。
- [ ] Owner/站点、资源策略、运行开关与审计都有独立页面。
- [ ] 域名重命名、站点级覆盖、Node 撤销/恢复都要求理由与确认。
- [ ] 审计不出现凭据、Cookie、客户内容或 Node 本机数据。

### 安全与表达

- [ ] 用户看不到 Cookie、token、内部端口、绝对路径或原始工具日志。
- [ ] 外部写入仍需显式确认，移动端配置不能绕过权限策略。
- [ ] 私有 Page、邮件和对话沿用同一登录与租户边界。
- [ ] 用户可见错误包含影响、恢复状态和下一步，而不是仅显示错误码。

## 14. 设计稿操作

打开 `prototype.html` 后：

1. 顶部切换“桌面客户端 / 移动端网页 / Cloud 客户门户 / Cloud 管理后台”。
2. 桌面左侧导航可切换对话记录、Workers、邮件、Online Pages、数据、渠道、技能、初始化和运行设置；不再设置独立“远程访问”。
3. Workers 列表可点击并切换桌面详情；没有“新建 Worker”。
4. 邮件列表可点击切换正文；页面不含转交、标记或回复动作。
5. 数据区可在底部切换 Sheet；点击字段或单元格可查看字段说明，并查看筛选、排序、每页条数与分页示例。
6. Online Pages 可切换全部、私有、公开；“查看”进入客户端内页面详情。
7. 渠道页的托管域名与自定义域名按钮可查看默认浏览器交接流程。
8. 移动端左上角打开侧边菜单；动态、对话、Worker、邮件和 Online Pages 各有独立列表，并显示分批加载状态。
9. 点击活动可进入 Worker 详情、对话详情、邮件详情或 Online Page 正文。
10. Cloud 客户门户可切换我的 PA、域名与访问、连接与安装、服务状态。
11. Cloud 管理后台可切换运行总览、Owner 与站点、资源策略、运行开关、审计记录。
