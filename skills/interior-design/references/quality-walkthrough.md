# 装修方案交付走查

每次修改空间、墙体、门窗或家具后都重新走查。概念模型也必须生活上可用；`好看` 不能覆盖明显冲突。把结果写入模型的 `qualityReview`，再运行 `cli.mjs audit`。存在 blocking finding 时不得生成或发布 Page。

## 1. 几何与碰撞

- 检查非地毯类家具是否互相穿插，柜门、抽屉和家电使用面是否被占用。
- 检查家具是否位于对应房间内，墙体、门窗与家具是否异常重叠。
- 门洞前后至少保留 0.8m 操作区；入户门、卧室门、卫生间门不得被柜体、床、桌椅挡住。
- 拆墙或新增洞口必须回到原始户型图标注位置、墙厚和不确定性；不得把栅格图识别当作结构结论。

## 2. 动线与使用净空

- 从入户到客餐厅、卧室、卫生间、厨房和阳台必须存在连续路径，不能穿过床、桌、沙发或柜体。
- 主要通道概念净宽优先不低于 0.9m，局部次要通道不低于 0.75m；不足时调整布局或明确风险。
- 餐椅后、床侧、衣柜前、厨房操作面、洗烘机前和卫生间洁具前保留实际使用空间。
- 门扇、抽屉、柜门和椅子拉出后的动态范围不能互相冲突。

## 3. 生活习惯与需求追踪

- 逐条对应用户需求：家庭人数、就餐人数、收纳量、办公、儿童/老人、宠物、家政、采光与隐私。
- 避免“有物件但不能用”：无落座距离的电视、拉不开的餐椅、够不到的柜体、无操作面的厨房、无晾晒/洗烘路径的阳台。
- 需求冲突时明确取舍，不静默牺牲用户已确认的高优先级需求。

## 4. 页面与标注

- 桌面与移动横屏分别检查：标签不遮住关键结构，控件不覆盖主体，所有文本不裁切、不重叠。
- 户型调整标注包含位置、范围、墙厚/尺寸来源和复核提示；隐藏标注后模型保持可理解。
- 原始图、调整标注、SU 设计稿、楼层和用户需求之间的切换必须保持唯一激活态并真实更新内容。
- 移动横屏保留标注显示/隐藏能力；可减少默认标签密度，但不能删除该能力。
- 宽度、净宽、墙厚和家具尺寸只在户型图调整标注层使用带两端刻度的建筑尺寸线；SU 设计稿不展示尺寸数值。
- 拆除对象必须按用户户型图的相邻空间与方向再次核对；同一图中有多个卧室时，不能只凭“次卧”等泛称定位。
- 挑高与局部二层方案必须核对保留挑空、楼梯抵达、护栏、净高、结构复核说明和两层连续动线。

## `qualityReview` 记录

```json
{
  "qualityReview": {
    "status": "passed",
    "checks": {
      "furnitureCollision": true,
      "doorClearance": true,
      "circulation": true,
      "useClearance": true,
      "lifestyleFit": true,
      "labelLayout": true,
      "desktopAndMobile": true
    },
    "requirementTrace": ["六人就餐", "入户到各空间动线", "主卧收纳"],
    "findings": []
  }
}
```

`findings` 可记录已修复或仍待用户/现场确认的问题；未解决的 blocking 项必须保留且 `status` 不得为 `passed`。
